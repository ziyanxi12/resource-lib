## 3. 性能极限分析

### 3.1 并发与性能关系

```
并发数    QPS      平均延迟    P99延迟
  100    125.65    778ms      978ms
  150     78.21   1828ms     2287ms
  200     42.12   4511ms     5070ms
```

**关键发现**:
- **100并发**: 最佳性能区间，QPS 达到 125.65
- **150并发**: 性能开始下降，QPS 降至 78.21，延迟升至 1.8s
- **200并发**: 性能急剧下降，QPS 仅 42.12，延迟达 4.5s（触发熔断）

### 3.2 性能衰减曲线

| 并发提升 | QPS 下降幅度 | 延迟增长幅度 |
|----------|-------------|-------------|
| 100 → 150 | -37.7% | +135% |
| 150 → 200 | -46.0% | +147% |

**分析**: 并发从 100 提升到 200 时，QPS 下降 66%，延迟增长 480%。性能衰减呈指数级，说明服务在高并发下存在明显的排队等待。

---

## 4. 性能瓶颈定位

### 4.1 瓶颈分析

根据测试数据，主要瓶颈可能在于：

1. **向量服务处理能力饱和**
   - 向量服务在并发请求下可能成为瓶颈
   - 建议：检查向量服务的并发处理配置

2. **数据库连接池限制**
   - SQLite 可能在高并发下存在锁竞争
   - 建议：生产环境使用 MySQL，并优化连接池

3. **HTTP 客户端连接数限制**
   - 高并发时连接复用效率下降
   - 建议：增加客户端连接池大小

4. **请求排队等待**
   - 并发超过处理能力时，请求在队列中等待
   - 建议：增加服务端 worker 数量

### 4.2 推荐配置

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| **最大并发数** | 100 | 安全范围内最高性能 |
| **预警并发数** | 150 | 性能下降，需监控 |
| **危险并发数** | 200+ | 延迟过高，避免使用 |

## 7. 优化建议

### 7.1 短期优化（1-2周）

1. **连接池优化**
   ```python
   # 增加数据库连接池
   SQLALCHEMY_POOL_SIZE = 20
   SQLALCHEMY_MAX_OVERFLOW = 40
   
   # 增加 HTTP 客户端连接池
   httpx.AsyncClient(
       limits=httpx.Limits(
           max_connections=200,
           max_keepalive_connections=50
       )
   )
   ```

2. **向量服务优化**
   - 增加向量服务 worker 数量
   - 开启批量搜索接口减少请求次数

3. **缓存层引入**
   - 对高频搜索词增加 Redis 缓存
   - 缓存 TTL 设置为 5-10 分钟

### 7.2 中期优化（1-2月）

1. **数据库升级**
   - 从 SQLite 迁移到 MySQL
   - 配置主从读写分离

2. **异步处理**
   - 使用消息队列（RabbitMQ/Kafka）削峰
   - 批量写入改为异步

3. **服务拆分**
   - 搜索服务独立部署
   - 向量服务独立扩容

### 7.3 长期优化（3-6月）

1. **分布式部署**
   - 多节点负载均衡
   - 向量库分片

2. **智能路由**
   - 根据搜索词热度路由到不同节点
   - 热点数据优先缓存
